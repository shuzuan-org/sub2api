# 邀请好友（Referral）功能方案

> 状态：已与用户确认范围并批准实施
> 设计稿：完整推荐返佣体系（专属邀请码、邀请链接、数据统计、邀请明细），但稿中多处标注「佣金规则可后续配置」

## 背景与范围

按设计稿新增「邀请好友」页面，入口放在侧边栏「个人资料」下方。

**本期范围（已确认收敛）：**

- **只做邀请框架 + 明细列表**。佣金计算、提现**不做**。
- 设计稿中「累计佣金」「可提取佣金」「充值人数」为**占位展示**（后端硬编码 0，前端标注「待配置」）。
- 奖励规则：**被邀请人注册即奖励** —— 新用户通过邀请码注册成功后，**邀请人**余额 +固定 U（金额由 settings 配置，默认 0，管理员可配）。
- 每个用户一个永久专属邀请码：**6 位，大写字母 + 数字**，固定绑定账号。

**关键技术约束：** 现有 `redeem_codes` 表是一次性消耗码、无 owner 列，**无法复用**，必须新建持久化推荐关系结构。

## 数据模型决策

在现有 `users` 表加 2 个可空列（最简、契合 ent、支持按码查 / 按邀请人列表 / 计数）：

| 列 | 类型 | 含义 |
|---|---|---|
| `referral_code` | `VARCHAR(6) UNIQUE` | 我的专属邀请码（懒创建） |
| `referred_by` | `BIGINT` | 我的邀请人 user_id |

不新建表 / edge。邀请码懒创建（首次访问邀请页时生成），唯一冲突重试 ≤5 次。

## 端到端验证（dockerized dev）

1. `cd backend && make generate && make wire`，编译
2. 重启 `sub2api-backend-src`；确认迁移 093 应用：`docker exec -i sub2api-postgres-src psql -U sub2api -d sub2api -c "\d users"` 含 `referral_code`、`referred_by`
3. 管理后台设 `invite_reward_amount` = `5`
4. 登录用户 A → `/invite` → 确认生成 6 位码；再次 GET 返回同一码；记录 A 余额
5. 无痕开 `/register?invite=<A_code>` → 注册用户 B
6. 验证 `select referred_by from users where email='B'` = A.id；A 余额 +5；重登 A / 查余额接口反映新值（缓存已失效）
7. A 的 `/invite` 列表显示 B（状态 已注册，累计充值 0）；按 B 邮箱搜索可过滤；分页正常

## 风险

- **ent/wire 必须重生成**：跳过 `make generate` / `make wire` → 新字段 / provider 编译失败
- **事务边界**：用户行由 `userRepo.Create` 先提交，奖励在**独立短事务**中尽力执行，与注册非原子（刻意设计）
- **6 位碰撞**：~1B 空间，唯一冲突必须重试；懒创建 `WHERE referral_code IS NULL` 保证并发首访幂等
- **无效邀请码绝不阻断注册**：所有 referral 查找 / 发放错误在 auth_service 中 log 并吞掉
- **缓存失效**：发放后必须失效邀请人 auth + billing 缓存，否则余额陈旧
- **占位字段清晰**：佣金 / 充值字段后端硬编码 0，UI 标注「待配置」，避免误导
